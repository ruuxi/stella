# Modifying Stella Desktop

`desktop/` is the Stella product itself. Most of the structure (TanStack
file-system routes, sidebar discovery, lazy views, Convex queries) is
self-evident from `ls desktop/src/` and reading any existing app folder.
This skill only documents the things that bite agents that *aren't*
obvious from the code.

## Adding a new app — use the scaffold

For "add a Notes / Foo page", run the
[`create-stella-app`](../create-stella-app/SKILL.md) scaffold and edit only the resulting
`<Component>View.tsx`. Don't hand-write metadata / route / shell files.

## Process model — pick the right one

The single biggest source of agent confusion. Stella runs as three
cooperating processes:


| Process                  | Code under                                  | Notes                                                                                      |
| ------------------------ | ------------------------------------------- | ------------------------------------------------------------------------------------------ |
| Main                     | `desktop/electron/`                         | Owns windows, IPC, native APIs, the canonical `UiState`, the runtime kernel. No DOM.       |
| Renderer (full shell)    | `desktop/src/` (entry: `main.tsx`)          | The full app UI. Browser DOM + React + Vite-bundled.                                       |
| Renderer (voice overlay) | `desktop/src/` (entry: `overlay-entry.tsx`) | Tiny always-on-top voice orb window. **Has no router** — reads `UiState` for active state. |


Renderer ↔ Main goes through `window.electronAPI.`* (preload bridge) →
IPC handlers under `desktop/electron/ipc/`. **Read
`desktop/STATE_OWNERSHIP.md` before changing anything that crosses the
boundary** — it's the source of truth for which process owns what.

## Non-obvious gotchas

- `desktop/src/routeTree.gen.ts` is auto-generated by Vite's TanStack
plugin on every save. Don't edit it; edit files under
`desktop/src/routes/` instead.
- The Sidebar discovers apps via `import.meta.glob("../../app/*/metadata.ts")`
in `desktop/src/shell/sidebar/Sidebar.tsx`. Don't hardcode entries
there.
- The voice overlay has no router — features that need to be reachable
from it must flow through `UiState`, not router state.
- `useFullShellChat()` is mounted exactly **once** in
`desktop/src/routes/__root.tsx` inside `<ChatRuntimeProvider>`.
Everything else (chat route, sidebars) reads it via
`useChatRuntime()`. Mounting it elsewhere double-instantiates the
streaming subscription.
- Tag interactive UI with `data-stella-label`, `data-stella-state`,
`data-stella-action` so `stella-ui` can drive it without reading
component trees.

## Validation

The root `package.json` owns Stella's desktop/runtime dependencies and scripts.
The right commands:

Run from the Stella install root (the directory containing `desktop/`,
`runtime/`, `state/`) so commands target the same tree the running app
loads from:

```bash
bunx --package typescript@5.9.3 tsc -p desktop/tsconfig.app.json --noEmit   # renderer
bun run electron:typecheck                                                 # main + preload
bun run lint
bun run test:run -- tests/runtime/sidebar-discovery.test.ts tests/runtime/route-smoke.test.ts
```

After scaffolding or editing any sidebar app, run at minimum:

```bash
bun run test:run -- tests/runtime/sidebar-discovery.test.ts tests/runtime/route-smoke.test.ts
```
